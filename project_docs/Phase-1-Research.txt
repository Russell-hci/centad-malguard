# Malware Image Representation and Datasets

Malware binaries (e.g. Windows PE files) are essentially byte sequences without inherent 2D structure. Converting them to images imposes a spatial layout that CNNs can exploit. For example, Nataraj *et al.* visualised entire executables as gray-scale images by mapping each byte to a pixel intensity【100†L169-L177】. They observed that malware from the same family yields visually similar textures and layouts in the image (see Figure 1 of [100]).  Typical conversions include: simple **grayscale mapping** (reading bytes row‑by‑row into a fixed-width image)【100†L169-L177】, **color-encoded byte‑class maps** (e.g. mapping printable ASCII vs control bytes to different color channels)【18†L341-L342】, and **space-filling curves** (reordering bytes via a Hilbert or spiral curve to preserve locality)【16†L962-L970】【104†L69-L78】. The simplest method – grayscale mapping – often works surprisingly well【16†L962-L970】. For instance, Saxena and Das note that when a PE file is reshaped into a 2D gray image, “section boundaries, padding, or encrypted payloads” become visible as image textures and blocks, which modern CNNs can readily learn【24†L128-L136】. In contrast, more complex encodings (Hilbert, entropy maps, etc.) attempt to enhance locality, but an empirical study found “the simplest (Grayscale) yielded the best results; several others were comparable”【16†L962-L970】. In practice then, grayscale or simple color mappings are common and effective. Importantly, these image representations have been shown to be fairly robust to obfuscation – obfuscated executables still produce similar broad image patterns【100†L169-L177】.

However, each conversion has trade‑offs. Grayscale images require no metadata but lose any structural ordering beyond linear. Space‑filling curves (Hilbert/spiral) can preserve some locality of nearby bytes【15†L8-L16】【104†L69-L78】, potentially grouping related instructions together, but add preprocessing complexity. Byte‑class coloring can highlight strings or code vs data blocks【18†L341-L342】, at the cost of extra channels. In sum, **grayscale images** are widely used (e.g. in the MALIMG dataset【100†L169-L177】) and feed directly into 2D CNNs, whereas more sophisticated mappings can be tried if justified by data.


### Why CNNs on Malware Images?

Treating malware as images allows leveraging decades of CV research. When viewed as images, *sections of the executable appear as coherent textures or blocks*, and repeated code patterns or instruction sequences form distinctive shapes【24†L128-L136】. CNNs excel at detecting such spatial patterns. As Saxena and Das explain, “modern CNNs are extremely good at recognizing such visual regularities” in the images【24†L128-L136】. Without any disassembly or manual feature engineering, a CNN can learn, for instance, that one texture corresponds to typical x86 code while another corresponds to data or packed sections. Early experiments bear this out: Nataraj *et al.* achieved ~98% accuracy using simple image features on 9.5K samples (25 families)【100†L169-L177】. Moreover, empirical work shows image-based models can generalize across obfuscations. For example, [100] notes resilience to section encryption. Even in presence of adversarial manipulations, models that learn *structural* cues (textures from code layout) remain effective【24†L128-L136】【24†L139-L147】. In summary, images turn malware classification into a vision problem where CNNs can automatically extract discriminative features, often outperforming hand‑crafted signatures【100†L169-L177】【24†L128-L136】.

### Common Malware Image Datasets

- **MalImg (Kaggle)**: A canonical dataset of malware images. It contains ~9.4K samples across 25 families (Adialer, Swizzor, etc.)【32†L58-L61】【100†L169-L177】. Each sample is the gray-scale image of the entire executable. Notably, in the Kaggle release these images are resized to 32×32【32†L58-L61】, which limits the detail available. *Strengths:* It is publicly available and covers many known families. *Weaknesses:* It only provides images (no raw bytes), so you can’t try alternate transformations (Hilbert, etc.) on the originals. The low resolution also truncates subtle patterns【12†L84-L91】. It serves well as a proof‑of‑concept but may overfit easily due to its small size. Many researchers use it for preliminary experiments, but results on MalImg alone may not generalize.

- **Microsoft Big 2015 (BIG2015)**: A huge Kaggle challenge dataset of ~10–11K Windows PE samples spanning 9 families (Ramnit, Lollipop, Kelihos, Vundo, Simda, Tracur, Kelihos, Obfuscator, Gatak)【35†L54-L62】. The raw data is hex-dumps (without PE headers) plus disassembly. *Strengths:* It’s large and contains raw content, so you can reconstruct byte images in multiple ways. *Weaknesses:* It has no benign class (only malware families), and its classes are highly imbalanced (e.g. only 42 samples of Obfuscator.AC vs thousands of others【35†L54-L62】). Converting it to images requires decoding hex to bytes and then padding/truncating (e.g. fixed 224×224 pixels as in some works). Because it’s so large (0.5 TB raw), training on the whole dataset with heavy models can be expensive. It is a standard benchmark (used in [35]) but does not include benign examples, limiting evaluation of false positives. Also, its static images may not reflect live behavior.

- **CIC-MalMem-2022 (CIC-MalMem)**: A memory-dump dataset from University of New Brunswick. It contains 58,596 samples (29,298 malicious vs 29,298 benign)【40†L150-L153】. The malware cover Spyware (AdClicker etc.), Trojans (Zeus, Dridex), and Ransomware (Conti), and benign samples are real Windows processes. *Strengths:* It is balanced and realistic (includes benign vs malware). The raw data are memory dumps which can be visualised (each dump can be treated as bytes for an image). It explicitly targets obfuscated malware: e.g. many families are known to use packing/encryption. *Weaknesses:* It’s much larger and more complex; memory dumps may include noise and mixture of processes, so classification by simple image patterns might be less direct. Also, as with BIG2015, images must be constructed from binary dumps (no ready-made images). Its focus on memory rather than file images means pre-processing (parsing memory sections) is needed.

**Dataset comparison summary:**

| Dataset     | Type                    | Size             | Classes                | Image Data  | Strengths                           | Weaknesses                          |
|------------|------------------------|------------------|------------------------|------------|------------------------------------|-------------------------------------|
| **Malimg** | Grayscale images       | ~9.4K samples    | 25 malware families    | Yes (32×32) | Easily accessible; classic benchmark【100†L169-L177】 | No raw binaries (cannot reprocess); low resolution; small scale |
| **BIG2015**| Raw PE (hex + disasm)  | ~10–11K samples  | 9 malware families     | No (hex given) | Large, diverse malware; raw bytes available | No benign class; class imbalance【35†L54-L62】; heavy to process |
| **CIC-MalMem-2022** | Raw memory dumps        | 58,596 samples   | Malware (3 categories, 15 families) + benign | No (binary dumps) | Balanced malware/benign; real-world processes【40†L150-L153】 | Complex (memory noise); images need careful construction |

Each dataset has limitations for a rigorous pipeline. Malimg’s small, pre-generated images make replication of experiments hard and raise generalization concerns【12†L84-L91】. BIG2015 is closer to raw data but lacks non-malware, so one cannot measure false positive rates easily. CIC-MalMem is realistic and balanced, but its focus on memory means image patterns may not align cleanly with file contents. In choosing a dataset, one must balance ease-of-use versus realism. In practice, a combination (e.g. Malimg or BIG2015 for initial development, CIC-MalMem for testing) may be justified to ensure results are scientifically sound.

# Lightweight Deep Learning Models

Constrained devices demand models with few parameters and low compute.  **MobileNetV3**, **ShuffleNetV2**, **EfficientNet-B0** and compact vision Transformers (like **MobileViT** or **TinyViT**) exemplify such architectures. They achieve efficiency through specific design patterns: *depthwise separable convolutions*, *bottleneck blocks*, *grouped convolutions*, *channel shuffling*, and *compound scaling*. For instance, MobileNetV3 (Howard *et al.*) uses separable convolutions and inverted residual blocks to dramatically cut operations【42†L126-L134】. ShuffleNetV2 (Ma *et al.*) uses pointwise group convolutions with a “channel shuffle” to equalize computation across threads【42†L136-L139】. EfficientNet (Tan *et al.*) uses a *compound scaling* rule to uniformly balance depth, width and input resolution for maximal accuracy per FLOP【42†L142-L148】. Modern light transformers like MobileViT inject a few self-attention modules at the end of a CNN backbone to capture global context without the huge overhead of a full ViT【48†L63-L70】. These ideas yield models with only a few million parameters and tiny compute costs, yet reasonable accuracy on ImageNet.

**Comparison of example models:**

| Model                | Params (≈)   | GFLOPs    | Strengths                                                      | Weaknesses                                        |
|----------------------|------------|---------|--------------------------------------------------------------|---------------------------------------------------|
| **MobileNetV3-Small**  | 2.54M【53†L146-L154】 | 0.06【53†L146-L154】 | Ultra-light (depthwise conv, inverted bottlenecks)【42†L126-L134】; fast inference on mobiles.     | Lower accuracy; limited capacity for complex tasks.   |
| **ShuffleNetV2 (1.0×)** | 2.28M【55†L151-L159】 | 0.14【55†L151-L159】 | Very efficient (channel shuffle, group conv)【42†L136-L139】; small memory footprint. | Accuracy suffers compared to larger nets.         |

| **MobileViT-Small**   | ~6M【48†L63-L70】    | ~0.8*  | Captures both local and global context (CNN+Transformer)【48†L63-L70】; outperforms MobileNet for similar size. | Heavier than pure CNNs; newer and less proven.    |

*: GFLOPs for MobileViT can vary by image size; the paper reports ~78.4% top-1 at 6M params【48†L63-L70】.

These model choices align with edge constraints. For example, MobileNetV3-Small with ~2.5M parameters and only 0.06 GFLOPs【53†L146-L154】 can run inference in milliseconds on a smartphone. ShuffleNetV2 similarly has only ~2.3M params【55†L151-L159】. In contrast, a standard ResNet50 has ~25M params and ~4.1 GFLOPs, which would be impractical on-device. As [42] notes, *“lightweight architectures… reduce memory and computational overhead while maintaining performance, enabling real-time inference on low-power devices”*【42†L69-L77】. In malware classification, where models might run on network appliances or endpoints, these small models allow deployment with limited CPU/GPU and memory.

However, the trade-off is accuracy and capacity. A very tiny model may underfit complex malware features. The project’s goal is not to maximize raw accuracy but to show feasibility with a small model. Thus architectures like EfficientNet-B0 or MobileViT (which are on the larger side of “lightweight”) can be tested as upper-bounds, while MobileNetV3 and ShuffleNetV2 serve as minimal baselines. The final model choice will hinge on balancing accuracy vs resource use for the target device.

# Adversarial Attacks and Threat Models

An *adversarial example* is an input modified by a small, intentional perturbation that causes misclassification【64†L53-L61】【76†L37-L42】. Formally, given a clean image $x$ and true label $y$, an adversary seeks $x' = x + \delta$ with $\|\delta\|\le\epsilon$ (for some norm bound $\epsilon$) such that the model predicts a wrong class $y'\neq y$ with high confidence【64†L53-L61】【76†L37-L42】. Goodfellow *et al.* define these as “worst-case perturbations… such that the perturbed input results in the model outputting an incorrect answer with high confidence”【64†L53-L61】. In practice, such perturbations on malware images might correspond to, e.g., appending bytes or flipping bits in non‑critical sections (no-ops) so that the resulting image is visually almost identical but classified differently. Despite being small, these changes can have **large security impact**: Grosse *et al.* showed that carefully crafting just a few feature changes caused a malware detector to misclassify 63% of malicious samples as benign【76†L19-L23】. Thus even slight perturbations can fully undermine trust in a classifier.

**Common attacks:** In a *white-box* scenario, the attacker knows the model parameters and can compute gradients. The classic FGSM (Fast Gradient Sign Method) attack by Goodfellow *et al.* perturbs the input in the direction of the sign of the loss gradient:
\[
x' = x + \epsilon\,\mathrm{sign}(\nabla_x L(\theta,x,y)) \,.
\]
This one-step method is very fast and often effective against undefended models. Iterative versions (Basic Iterative Method, BIM) apply FGSM repeatedly; Madry *et al.* formalized this as *Projected Gradient Descent* (PGD): repeatedly update
$$x^{t+1} = \Pi_{x+S}\bigl(x^t + \alpha\,\mathrm{sign}(\nabla_x L)\bigr),$$
projecting back into the allowed $\ell_\infty$ ball【69†L232-L239】. PGD is essentially the “strongest” first-order (gradient-based) attack: it maximizes the loss under an $\ell_\infty$ budget. For instance, Madry *et al.* explicitly identify PGD as the adversary for $\ell_\infty$ and discuss its iterative update【69†L232-L239】.

Stronger attacks like the Carlini–Wagner (C&W) methods solve a dedicated optimization to minimize the perturbation while changing the label【71†L53-L61】. C&W can target any norm ($L_2, L_0,$ etc.) and often succeed where FGSM/PGD fail, even breaking defenses like distillation【71†L53-L61】. For example, Carlini & Wagner show that for any input $x$ and target class $t$, one can find $x'\approx x$ classified as $t$, which “makes it difficult to apply neural networks in security-critical areas”【71†L53-L61】. In black-box scenarios (no gradient access), attackers typically exploit *transferability*: adversarial examples crafted on one model often fool another【64†L58-L64】. Goodfellow’s work notes this cross-model generalization as a curious feature of adversarial examples【64†L58-L64】. In practice, a black-box attacker might train a substitute model (perhaps on surrogate malware images) and transfer crafted examples.

The **cost** of attacks varies. FGSM is very cheap (one gradient step), while PGD/BIM (with dozens of iterations) is more expensive. C&W attacks use iterative optimization with hundreds of gradient steps. All are feasible off-line (at analysis time), but on-line attacks (e.g. query-based) may resort to fewer queries. However, even modest attacks suffice: for example, Reilly *et al.* demonstrated that a single PGD attack could drop a CNN’s accuracy from ~95% to ~4.5% on malware images【95†L79-L85】, at which point even limited adversarial training doubled accuracy back up. In cybersecurity, such vulnerabilities are critical: an attacker could hide malware by altering only a few bytes (e.g. benign padding) to evade detection, undermining system reliability.

# Defenses for Lightweight Models

Adversarial defenses aim to increase a model’s **robust accuracy** (the accuracy under attack). The most important practical defense is **adversarial training**: augment training with adversarial examples (e.g. PGD-crafted) and optimize for worst-case loss【84†L30-L34】. Madry *et al.* showed that solving a robust optimization (minimize worst-case loss over $\|\delta\|_\infty\le\epsilon$) yields models with “significantly improved resistance to a wide range of attacks”【84†L30-L34】. In practice, this means at each batch computing adversarial versions (via FGSM or PGD) and training the model on them. For example, Reilly *et al.* added GAN-generated adversarial malware images to the training set; this nearly *doubled* the model’s accuracy under PGD attack (from ~4.5% to ~9% in their experiment)【95†L79-L85】. The drawback is cost: adversarial training roughly multiplies training time by the number of attack steps used (PGD-10 means 10× slower than natural training), and the model may grow larger to accommodate robustness.

**Augmentation and input transforms:** Simpler defenses include random data augmentation (flips, crops, noise) and input preprocessing. For images, techniques like JPEG compression or bit-depth reduction can sometimes neutralize small perturbations. Denoising networks or feature squeezing (e.g. reducing color bit-depth) have been proposed, but they often reduce accuracy or can be circumvented. Importantly, any input transform must preserve the semantic meaning of a malware image; in practice, most perturbations must correspond to valid byte changes.

**Randomized smoothing:** A promising certified defense is *randomized smoothing*【79†L48-L56】. Here, one adds Gaussian noise to the input at inference and takes a majority vote over many noisy copies. This yields a new “smoothed” classifier which is provably robust within a certain $\ell_2$ radius. For instance, Cohen *et al.* certify that their smoothed ImageNet model achieves ~49% accuracy under any $L_2$ perturbation of norm ≤0.5【79†L48-L56】. The upside is a guaranteed robustness bound; the downside is inference cost (potentially hundreds of forward passes per input) and the fact that certified radius shrinks with higher-dimensional images. For lightweight models, smoothing could be applied but would likely require lighter noise (since $\ell_2$ balls scale with dimension) and increase latency.

**Other methods:** Distillation (Papernot *et al.*) once was considered, but was later broken by C&W. More recent ideas include ensemble methods and trusted input transformations. Broadly, we find that *adversarial training (or equivalents like adding GAN-perturbed data)* is the most practical for moderate model sizes【84†L30-L34】【95†L79-L85】. Data augmentation with adversarial examples (or using generated adversarial samples in GANs) shows real benefit【95†L79-L85】. Techniques like randomized smoothing provide theoretical guarantees but at heavy cost. Since our focus is on a small model for deployment, we anticipate using adversarial training (e.g. PGD during training) as the core defense, perhaps augmented by simple augmentation and a small amount of input denoising. More exotic defenses (certified models) likely exceed our deployment budget.

# Explainable AI (XAI)

In malware classification, interpretability is crucial: security analysts must trust *why* a model flags something as malicious. Visual explanation tools like Grad-CAM help inspect the model’s “attention” on image inputs. Grad-CAM produces a heatmap over the image by taking gradients of the target class score w.r.t. the last convolutional feature maps. For CNN-based malware detectors, this highlights which parts of the byte image contributed most to the decision. For example, a Grad-CAM map might light up a region corresponding to a suspicious code segment or embedded string, indicating that the model is focusing on real malicious patterns. As one recent study notes, such visualisations “provide critical explainability by highlighting malicious regions… enabling security analysts to validate detection logic”【87†L65-L73】.

Critically, explainability allows us to check model reasoning, especially under attack. If an adversarial example changes the model’s focus from a meaningful region (say, the code section of an executable) to an irrelevant patch, that indicates the model is relying on fragile cues. By comparing Grad-CAM on clean vs adversarial malware images, we can see if the perturbation causes a drastic shift in attention. In this way, XAI is not an optional add-on but a diagnostic tool. It lets us catch cases where the model’s accuracy might still be high, but for the wrong reasons. For lightweight models in particular, Grad-CAM is especially suitable: it is designed for CNNs, has low overhead, and produces intuitive visual overlays.

In summary, we will integrate explainability into our evaluation: for each detected (or misdetected) malware image, we will generate Grad-CAM maps to illustrate which pixels (hence which byte regions) drove the decision. This helps ensure the model isn’t basing its classification on spurious patterns, and it can reveal how defenses alter the decision process. In this way, interpretability becomes part of validating robustness and trustworthiness in our pipeline【87†L65-L73】.

From these and other studies, we note recurring gaps:

- **Focus on accuracy, not robustness:** Many works (e.g. [100], O’Shaughnessy [104]) report high clean accuracy but do not consider adversarial attacks at all. Conversely, adversarial-ML works on malware often use non-image features (like Grosse【76】) or Android-specific features, not images.
- **Heavy models vs deployment:** Most image-based classifiers use large CNNs (ResNet50, VGG) or traditional ML on extracted features. Few consider lightweight models suited to edge. Our survey found *no prior work that evaluates MobileNet/EfficientNet/etc specifically on malware images*.
- **Explainability:** Virtually no work has applied XAI to malware images. (The Atlantis study【87】 uses Grad-CAM for encrypted traffic, which is promising, but image-based classification specifically has not been paired with explanation.)
- **Combined requirements:** Crucially, *no existing study combines*: (1) **malware image classification**, (2) **lightweight architecture**, (3) **robust adversarial evaluation**, and (4) **explainability**.  Our project’s novelty is to integrate all four. For example, Reilly et al. handle (1) and (3) with a heavyweight ResNet and GAN augmentation, but ignore model size and explainability. O’Shaughnessy addresses (1) in a static+dynamic hybrid but no adversaries. SAXena et al. mention CNNs on malware images【24†L128-L136】 but do not consider lightweight constraints or adversarial robustness.

**Research gap:** There is a gap in the literature for a coherent pipeline that applies *compact CNNs* (or ViTs) to malware images **and** rigorously evaluates both clean and robust accuracy **while also providing interpretable visual explanations**. Our project fills this niche: it will justify using grayscale or space-filling images (as supported by [24,16,104]) and then systematically compare lightweight models on those images, including attack/defense tests (FGSM, PGD) much like Reilly【95†L79-L85】 but for small nets, and overlay Grad-CAM maps to inspect them as in [87].

**Conclusion and Recommendations:** In summary, the literature supports our planned choices: *malware-to-image* conversions (grayscale or Hilbert) are effective【24†L128-L136】【16†L962-L970】; *lightweight nets* like MobileNetV3/ShuffleNetV2/EfficientNet-B0 offer a strong accuracy‐vs‐size trade-off【42†L126-L134】【53†L146-L154】; *adversarial training* (with PGD) is the most promising defense【84†L30-L34】; and *Grad-CAM* will be used to interpret model focus【87†L65-L73】. We will implement these in PyTorch (using **torchvision** and **timm** for models, **TorchAttacks** or custom code for attacks, and Grad-CAM libraries) to build a scientifically robust malware image classifier tailored for edge deployment.
